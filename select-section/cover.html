Add Item to Section
Option C selected  ·  Last audited: April 28, 2026
Problem Statement

When users click "+ Add Item" in a Roofr Proposal, the item is always appended to the bottom of the full list. Getting it into a specific section requires drag-and-drop across potentially many rows — significant friction for users building multi-section proposals. Option C solves this with two complementary paths: a per-section + button in the Actions column for zero-click direct adds, and a section picker on the global footer button as a fallback when the target section is off-screen.

Decided — specs locked
🔵
Single section — skip picker, insert directly
When only one section exists, the global "Add item to section" button bypasses the picker entirely and inserts the new row directly into that section.
🔵
Hidden section — added items also hidden
Adding an item to a hidden section inherits is_visible: false. The section stays selectable in the picker and its + button stays enabled — items just land hidden.
🔵
Long section name — truncate at 40 characters
Section names are capped at 40 characters in both the section header row and the picker option. Overflow is truncated with ellipsis.
🔵
Item save failure — error toast
If the API save fails after an item is committed, show an error toast with copy explaining why and prompting the user to try again. The row stays in place for retry.
States Still Needed
🔴 High priority — will appear in production
Picker overflow — many sections
No max-height or scroll on the picker. With 6+ sections it will clip or overflow off the top of the viewport. Proposals with many headings will hit this immediately.
Source: option-c.html line 291–305 — no overflow or max-height · Real: HeadingGroup[] has no count limit
⚪ Low priority / nice to have
Picker count decrement on delete
The prototype only increments counts. In production, count should be driven off HeadingGroup.items.length dynamically so deletes are reflected.
Singular / plural copy
"1 item" vs "2 items" in the picker count badge — should use an i18n plural key consistent with the rest of the app.
Undo after commit
Escape during inline editing removes the row, but after committing a name there is no undo. Not unique to this feature but worth noting for handoff.
